Définition : Une liste est mutable (modifiable), un tuple est immutable (non modifiable).
Exemple : liste = [1, 2, 3] peut être modifiée avec liste.append(4), alors que tuple = (1, 2, 3) ne peut pas être modifié.
Cas d'usage : Utilisez un tuple pour des données fixes (coordonnées GPS, configuration), une liste pour des données dynamiques (panier d'achat, historique).
Performance : Les tuples sont plus rapides et prennent moins de mémoire car ils sont optimisés pour l'immutabilité.
Définition : Un décorateur est une fonction qui prend une fonction en paramètre et retourne une fonction modifiée. Il permet d'ajouter des fonctionnalités sans modifier le code original.
Exemple : @login_required vérifie l'authentification avant d'exécuter la vue. Syntaxe : @decorator avant la définition de fonction.
Cas d'usage : Logging, timing, authentification, validation, caching. Très utilisé dans Django pour les vues.
Avantage : Réutilisabilité et séparation des préoccupations (concerns).
Définition : == compare les valeurs, is compare l'identité en mémoire (même objet).
Exemple : a = [1,2,3] et b = [1,2,3] → a == b est True mais a is b est False (objets différents).
Cas d'usage : Utilisez is pour comparer à None (if x is None), == pour comparer les valeurs.
Attention : Les petits entiers (-5 à 256) sont internés, donc a = 5; b = 5; a is b peut être True.
Définition : Syntaxe concise pour créer des listes à partir d'itérables avec une condition optionnelle.
Exemple : [i**2 for i in range(10) if i % 2 == 0] crée une liste des carrés des nombres pairs de 0 à 9.
Avantages : Plus lisible et souvent plus rapide qu'une boucle for classique.
Variantes : Dict comprehension {k: v for k, v in items}, set comprehension {x for x in range(10)}.
Définition : Structure try/except/else/finally pour gérer les erreurs de manière contrôlée.
Exemple : try: result = 10/0 puis except ZeroDivisionError: print("Division par zéro").
Bonnes pratiques : Capturer des exceptions spécifiques (pas except: seul), utiliser else pour le code qui ne doit pas lever d'exception, finally pour le nettoyage.
Cas d'usage : Gestion de fichiers, appels API, validation de données. Toujours logger l'erreur pour le debugging.
Définition : Verrou global qui empêche plusieurs threads Python d'exécuter du bytecode simultanément sur le même processus.
Impact : Le multithreading Python n'est pas vraiment parallèle pour le CPU-bound (calculs intensifs).
Solution : Utiliser multiprocessing pour le vrai parallélisme, ou des bibliothèques comme NumPy qui libèrent le GIL.
Cas d'usage : Le GIL n'est pas un problème pour l'I/O-bound (requêtes réseau, fichiers) car le GIL est libéré pendant les opérations I/O.
Définition : *args collecte les arguments positionnels variables en tuple, **kwargs collecte les arguments nommés variables en dictionnaire.
Exemple : def func(a, *args, **kwargs): peut être appelée avec func(1, 2, 3, x=4, y=5) où args=(2,3) et kwargs={'x':4, 'y':5}.
Cas d'usage : Fonctions wrapper, décorateurs, fonctions avec nombre variable d'arguments. Très utile pour passer des arguments à des fonctions parentes.
Définition : Fonction qui utilise yield au lieu de return, produisant des valeurs à la demande (lazy evaluation).
Exemple : def gen(): yield 1; yield 2 crée un générateur qui produit 1 puis 2 à chaque appel de next().
Avantages : Économie mémoire (pas besoin de stocker toute la séquence), peut être infini, évaluation paresseuse.
Cas d'usage : Traitement de gros fichiers ligne par ligne, séquences infinies, pipelines de données. Utilisé dans Django pour les QuerySets.
Définition : "Si ça marche comme un canard et ça fait le bruit d'un canard, c'est un canard" - On vérifie les méthodes disponibles, pas le type exact.
Exemple : Si un objet a une méthode read(), on peut l'utiliser comme un fichier, peu importe son type réel.
Avantage : Code plus flexible et réutilisable, moins de couplage avec les types spécifiques.
Python : Le duck typing est au cœur de Python - les protocoles (comme le protocole itérable) sont basés sur les méthodes, pas les types.
Définition : Méthode d'instance prend self (accès à l'instance), méthode de classe prend cls avec @classmethod (accès à la classe), méthode statique avec @staticmethod (ni self ni cls).
Exemple : @classmethod pour créer des instances alternatives, @staticmethod pour des fonctions utilitaires liées à la classe.
Cas d'usage : Factory methods avec @classmethod, fonctions utilitaires avec @staticmethod, logique métier avec méthodes d'instance.
Définition : Objet qui gère automatiquement l'acquisition et la libération de ressources via les méthodes __enter__ et __exit__.
Exemple : with open('file.txt') as f: ouvre le fichier et le ferme automatiquement, même en cas d'erreur.
Avantages : Code plus propre, gestion automatique des erreurs, pas d'oubli de fermeture de ressources.
Cas d'usage : Fichiers, connexions DB, verrous (locks), transactions. Utilisé partout en Python moderne.
Définition : Mutable = modifiable après création (list, dict, set), Immutable = non modifiable (tuple, str, int, frozenset).
Exemple : liste[0] = 5 fonctionne, mais tuple[0] = 5 lève une erreur.
Avantages immutabilité : Thread-safe, hashable (utilisable comme clé de dict), prévisible.
Cas d'usage : Utilisez des tuples pour des données fixes (clés de dict, coordonnées), des listes pour des données dynamiques.
Définition : Système automatique de gestion mémoire utilisant le comptage de références + détection de cycles.
Mécanisme : Chaque objet a un compteur de références. Quand il atteint 0, l'objet est supprimé. Un algorithme détecte les cycles de références.
Avantages : Pas besoin de libérer manuellement la mémoire, pas de fuites mémoire classiques.
Cas d'usage : Fonctionne automatiquement. On peut forcer avec gc.collect() si nécessaire pour les cycles complexes.
Définition : Shallow copy = copie les références (objets imbriqués partagés), Deep copy = copie récursive complète (objets indépendants).
Exemple : import copy; shallow = copy.copy(liste); deep = copy.deepcopy(liste)
Cas d'usage : Shallow copy pour des structures simples, deep copy quand vous voulez modifier les objets imbriqués indépendamment.
Attention : Les shallow copies peuvent causer des bugs subtils si vous modifiez des objets imbriqués sans le réaliser.
Définition : MVT = Model-View-Template (variante de MVC). Model = données (ORM), View = logique métier (contrôleur), Template = présentation (HTML).
Flux : Requête → URL → View (logique) → Model (données) → Template (rendu) → Réponse.
Avantages : Séparation claire des responsabilités, code maintenable, réutilisabilité des composants.
Cas d'usage : Dans un projet Django, les models définissent la structure DB, les views contiennent la logique, les templates affichent les données.
Définition : Object-Relational Mapping - Manipule la base de données avec du code Python au lieu de SQL brut.
Exemple : User.objects.filter(email='test@example.com') au lieu de SELECT * FROM users WHERE email='test@example.com'.
Avantages : Protection contre les injections SQL, portabilité entre SGBD, code plus lisible, migrations automatiques.
Cas d'usage : Utilisé partout dans Django. Permet de changer de SGBD (PostgreSQL → MySQL) sans modifier le code.
Définition : Fichier Python décrivant les changements de schéma de base de données (création/modification de tables, colonnes).
Processus : python manage.py makemigrations crée les fichiers, python manage.py migrate les applique.
Avantages : Versioning du schéma DB, réversibilité, collaboration en équipe, déploiement automatisé.
Cas d'usage : Chaque modification de model génère une migration. Les migrations sont commitées dans Git avec le code.

Définition : Problème de performance : 1 requête pour récupérer N objets, puis N requêtes supplémentaires pour leurs relations.
Exemple : for user in User.objects.all(): print(user.profile.bio) fait 1 requête pour users + N requêtes pour profiles.
Solution : select_related() pour ForeignKey (JOIN SQL), prefetch_related() pour ManyToMany (2 requêtes optimisées).
Cas d'usage : Toujours utiliser select_related ou prefetch_related quand vous accédez à des relations dans une boucle.

Encapsulation : Utiliser private pour les champs, public pour les méthodes (getters/setters).

Définition : == compare les références (adresses mémoire), .equals() compare le contenu des objets.
Exemple : String a = new String("test"); String b = new String("test"); → a == b est false (objets différents), mais a.equals(b) est true (même contenu).
Règle : Pour les primitifs, utilisez ==. Pour les objets, utilisez .equals() (sauf si vous voulez comparer les références).
Attention : Toujours redéfinir hashCode() si vous redéfinissez equals().
Définition : Interface = contrat pur (toutes méthodes abstraites), Classe abstraite = peut avoir du code concret et abstrait.
Différences : Interface = héritage multiple possible, pas de constructeur, méthodes par défaut public abstract. Classe abstraite = héritage simple, constructeur possible, méthodes concrètes possibles.
Cas d'usage : Interface pour définir un contrat (comportement), classe abstraite pour partager du code commun entre classes liées.
Exemple : List est une interface, AbstractList est une classe abstraite qui implémente List.
Définition : Framework de collections pour stocker et manipuler des groupes d'objets.

Map : Clé-valeur, pas de doublons de clés (HashMap, TreeMap, LinkedHashMap).
Cas d'usage : ArrayList pour listes dynamiques, HashSet pour éviter doublons, HashMap pour recherche rapide par clé.

Exemple : Dans une boucle, utilisez StringBuilder : StringBuilder sb = new StringBuilder(); sb.append("text");

Taux de Conversion

Formule : Taux de Conversion (%) = (Nombre de conversions / Nombre de visiteurs) × 100
Exemple : 100 conversions / 5000 visiteurs = 2%
Objectif : E-commerce : 2-3%, SaaS : 1-2%, Landing page : 5-10%

Panier Moyen

Formule : Panier Moyen = Chiffre d'affaires total / Nombre de commandes
Exemple : 50 000€ / 500 commandes = 100€
Objectif : Augmenter via upselling, cross-selling, bundling

Taux d'Abandon de Panier

Formule : Taux d'Abandon (%) = (Paniers abandonnés / Paniers créés) × 100
Exemple : 300 paniers abandonnés / 500 paniers créés = 60%
Objectif : Réduire sous 70% (moyenne e-commerce : 70-80%)

Revenue per Visit (RPV)

Formule : RPV = Chiffre d'affaires total / Nombre de visites
Exemple : 50 000€ / 10 000 visites = 5€ par visite
Objectif : Maximiser via optimisation conversion et panier moyen

Taux de Rebond

Formule : Taux de Rebond (%) = (Sessions à une seule page / Total des sessions) × 100
Exemple : 2000 sessions rebond / 5000 sessions = 40%
Objectif : E-commerce : < 50%, Blog : < 70%, Landing page : < 30%

LTV (Lifetime Value)

Formule simple : LTV = Revenu moyen par transaction × Nombre de transactions par an × Durée de vie moyenne (années)
Formule avancée : LTV = (Panier moyen × Fréquence d'achat) / Taux de désabonnement
Exemple : 50€ × 4 achats/an × 3 ans = 600€
Exemple avancé : (50€ × 4) / 0.33 (33% de désabonnement/an) = 606€
Objectif : LTV > 3× CAC (idéalement 5×)

CAC (Customer Acquisition Cost)

Formule : CAC = (Coûts marketing + Coûts ventes) / Nombre de nouveaux clients acquis
Exemple : (10 000€ marketing + 5 000€ ventes) / 100 clients = 150€
Objectif : CAC < LTV/3 (si LTV = 600€, CAC < 200€)

Ratio LTV/CAC

Formule : Ratio LTV/CAC = LTV / CAC
Exemple : 600€ / 150€ = 4:1
Objectif : > 3:1 (sain), > 5:1 (excellent), < 1:1 (non viable)

Payback Period (Période de Récupération)

Formule : Payback Period (mois) = CAC / (Revenu mensuel moyen par client)
Exemple : 150€ / 50€/mois = 3 mois
Objectif : < 12 mois (idéalement < 6 mois)

MRR (Monthly Recurring Revenue)

Formule : MRR = Nombre de clients payants × Prix moyen mensuel
Exemple : 1000 clients × 50€/mois = 50 000€ MRR
Variations : New MRR (nouveaux clients), Expansion MRR (upgrades), Churned MRR (perdus)

Time to First Byte (TTFB)

Définition : Temps entre la requête HTTP et la réception du premier octet de réponse
Objectif : < 200ms (excellent), < 500ms (bon), < 800ms (acceptable)
Impact : Chaque 100ms de réduction = +1% conversion (Amazon)

Largest Contentful Paint (LCP)

Définition : Temps de chargement du plus grand élément visible de la page
Objectif : < 2.5s (bon), 2.5s - 4s (à améliorer), > 4s (mauvais)
Impact : Facteur de ranking SEO, impact direct sur les conversions

First Input Delay (FID)

Définition : Temps entre l'interaction utilisateur et la réponse du navigateur
Objectif : < 100ms (bon), 100ms - 300ms (à améliorer), > 300ms (mauvais)
Impact : Expérience utilisateur, taux de rebond

Cumulative Layout Shift (CLS)

Définition : Mesure de la stabilité visuelle (décalages de contenu)
Formule : CLS = Σ (Impact fraction × Distance fraction)
Objectif : < 0.1 (bon), 0.1 - 0.25 (à améliorer), > 0.25 (mauvais)
Impact : Expérience utilisateur, taux de conversion

Page Speed Score

Définition : Score Lighthouse (0-100) basé sur les performances
Objectif : > 90 (excellent), 70-90 (bon), 50-70 (à améliorer), < 50 (mauvais)
Impact : SEO, conversions, expérience utilisateur

Taux de Clic Organique (CTR)

Formule : CTR (%) = (Clics organiques / Impressions) × 100
Exemple : 1000 clics / 50 000 impressions = 2%
Objectif : Position 1 : 30-40%, Position 2-3 : 15-20%, Position 4-10 : 5-10%

Trafic Organique

Formule : Trafic Organique = Σ (Clics depuis Google pour tous les mots-clés)
Métriques : Sessions organiques, Pages vues, Durée moyenne, Taux de rebond
Objectif : Croissance mensuelle de 10-20%

Position Moyenne

Formule : Position Moyenne = Σ (Position × Impressions) / Σ Impressions
Exemple : (1×1000 + 5×500 + 10×200) / 1700 = 3.5
Objectif : < 10 pour les mots-clés principaux

Taux de Conversion Organique

Formule : Taux de Conversion Organique (%) = (Conversions organiques / Sessions organiques) × 100
Exemple : 100 conversions / 5000 sessions = 2%
Objectif : Comparer avec autres sources (généralement meilleur que paid)

Formule :

ICE Score = (Impact + Confiance + Effort) / 3
Chaque critère noté de 1 à 10 :

📊 Impact (1-10)

Question : Quel est l'impact potentiel sur la métrique cible ?
Exemples : 10 = Impact majeur (doubler les conversions), 5 = Impact moyen, 1 = Impact faible

🎯 Confiance (1-10)

Exemples : 10 = Effort minimal (1 jour), 5 = Effort moyen (1 semaine), 1 = Effort maximal (1 mois)

  • Confiance : 7/10 (données A/B test similaires)
  • Effort : 9/10 (changement simple, 1 jour)
ICE Score : (8 + 7 + 9) / 3 = 8.0

💡 Utilisation : Prioriser les expériences avec le score ICE le plus élevé pour maximiser l'impact avec le moins d'effort.
